# Refactoring Brief: Replace `metal-cpp` and `metal-docs.json` with `metal-ast.json`

## Background

The current `Metal.NET.Generator` relies on two input sources:

1. **`metal-cpp`** — A directory of C++ header files, parsed by `CppParser` via regex to extract enums, classes, methods, free functions, structs, block type aliases, and ObjC selector mappings.
2. **`metal-docs.json`** — A documentation database represented by `DocDatabase` models, providing summaries, deprecation info (deprecated/deprecationMessage), and group-categorized member information.

Parsed results are aggregated into `GeneratorContext`, then consumed by `CSharpEmitter` together with `TypeMapper` and the doc database to produce final C# binding code.

## Refactoring Goal

Use **`metal-ast.json`** — generated by GitHub Actions via clang AST dump — as the **sole data source**, replacing both inputs above. `metal-ast.json` already contains all necessary information:

| Current Source | Corresponding `metal-ast.json` Field |
|---|---|
| `CppParser` → Enum definitions | `enums[]` (with `kind`, `name`, `underlyingType`, `isOptions`, `members[]`, `framework`) |
| `CppParser` → Class/Protocol definitions | `classes[]` + `protocols[]` (with `methods[]`, `properties[]`, `protocols[]`, `framework`) |
| `CppParser` → Struct definitions | `structs[]` |
| `CppParser` → Free functions | `functions[]` |
| `CppParser` → Typedef/Block types | `typedefs[]` |
| `CppParser` → Selector mappings | `methods[].selector` provided directly |
| `DocDatabase` → Deprecation info | `deprecated` / `deprecationMessage` fields on each entity and member |
| `DocDatabase` → Grouping/Sorting | **No longer needed** — the order in `metal-ast.json` is the final order |

## Key Changes

1. **Remove `CppParser.cs`** — Regex-based C++ header parsing is no longer needed.
2. **Remove `DocDatabase.cs`** — Deprecation/summary info is now embedded in `metal-ast.json`.
3. **Add `AstJsonParser.cs`** (or similar) — Deserialize `metal-ast.json` and map it to the existing `GeneratorContext` models (`EnumDef`, `ClassDef`, `MethodInfo`, `StructDef`, `FreeFunctionDef`, etc.).
4. **Update `Models.cs`** — Extend or adjust model definitions to accommodate `metal-ast.json` fields (e.g., `selector`, `returnNullability`, `nullability`, `framework`, `deprecated`).
5. **Update `GeneratorContext.cs`** — Remove `SelectorMap` and `RegisteredClasses` (selectors are now provided with each method); potentially add a `Protocols` list.
6. **Update `Generator.cs` / `Program.cs`** — Change entry point to accept a `metal-ast.json` file path instead of a `metal-cpp` directory path; remove `LoadDocDatabase` logic.
7. **Update `CSharpEmitter.cs`** — Remove dependency on `docDb`; deprecation info is read directly from models; the `framework` field can replace `CppNamespace` for determining output subdirectories and prefixes; remove grouping/sorting logic (output in JSON order).
8. **Update `TypeMapper.cs`** — ObjC-to-C# type mapping may need adaptation based on how types are represented in the AST (e.g., `MTLDevice *` vs `MTL::Device*`).

## Core Principles

- **Output structure remains unchanged** — Generated C# file organization (framework-based subdirectories, consolidated enum files, individual class files, etc.) stays identical to the current output.
- **No grouping or sorting required** — The data order in `metal-ast.json` is the generation order.
- **Single source of truth** — All type information, method signatures, properties, and deprecation markers come from `metal-ast.json`; no external header files or documentation JSON are needed.
